## 2025-02-27 - [XSS in inline event handlers]
**Vulnerability:** XSS inside the string literal of an inline event handler in index.html (onclick="...").
**Learning:** HTML entity decoding occurs before inline JavaScript executes in an attribute. Therefore `&#039;` is decoded back into `'` and can break out of a JS string literal, making `escapeHtml` insufficient for defending against XSS in this context.
**Prevention:** Avoid dynamic interpolations within inline JS attributes. Instead store data using `data-*` attributes and retrieve them safely via `this.getAttribute(...)`.
## 2024-05-27 - XSS via Federated API Payloads
**Vulnerability:** A URI-based XSS vulnerability existed where dynamically generated `href` and `src` attributes in `index.html` and `player.html` parsed unsanitized URLs from federated API sources (like instance URLs, profile avatars, and track covers). If an attacker provided a payload like `javascript:alert(1)`, it would bypass HTML escaping and execute in the context of the user's browser.
**Learning:** HTML escaping alone (`escapeHtml`) is insufficient for URLs. Attackers can provide malicious URI schemes (`javascript:`, `vbscript:`) that execute JS when interacted with, or in some contexts, when loaded. Additionally, `data:` URIs are legitimately required for some elements (like covers and avatars) but can be abused if not context-aware.
**Prevention:** Always validate and neutralize the URL scheme before injecting it into the DOM. Use a robust `sanitizeUrl` wrapper that relies on the native `new URL()` parser to enforce safe protocols (`http:`, `https:`) and conditionally blocks/allows `data:` URIs based on whether the context is an image `src`.
