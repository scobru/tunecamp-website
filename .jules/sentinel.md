## 2025-02-27 - [XSS in inline event handlers]
**Vulnerability:** XSS inside the string literal of an inline event handler in index.html (onclick="...").
**Learning:** HTML entity decoding occurs before inline JavaScript executes in an attribute. Therefore `&#039;` is decoded back into `'` and can break out of a JS string literal, making `escapeHtml` insufficient for defending against XSS in this context.
**Prevention:** Avoid dynamic interpolations within inline JS attributes. Instead store data using `data-*` attributes and retrieve them safely via `this.getAttribute(...)`.
## 2024-05-24 - Fix URI-based XSS in Federated Inputs
**Vulnerability:** External inputs from federated APIs (such as track covers, profile pictures, and instance URLs) were being placed directly into `href` and `src` attributes with only HTML escaping (`escapeHtml`). This allowed URI-based XSS if an attacker supplied a `javascript:` or `vbscript:` payload as a URL.
**Learning:** HTML escaping alone is insufficient for securing URL attributes (`href`, `src`). An attacker can use `javascript:alert(1)` which doesn't require `<` or `>` characters. However, sanitization must be flexible enough to still allow `data:` URIs specifically for image resources, as the Tunecamp network heavily utilizes them for base64-encoded cover art.
**Prevention:** Always validate and sanitize the URL protocol for dynamic attributes (blocking `javascript:`/`vbscript:` and conditionally allowing `data:`). This sanitized output MUST then be HTML-escaped (`escapeHtml(sanitizeUrl(url))`) before DOM insertion to achieve true defense in depth against both URI attacks and attribute breaking.
